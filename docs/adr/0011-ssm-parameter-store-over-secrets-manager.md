# ADR-0011: SSM Parameter Store over Secrets Manager for runtime credentials

## Status
Accepted

## Date
2026-05-04

## Context
The platform stores five runtime credentials in AWS:

- `anthropic-api-key` — used by the enrichment Lambdas and the agent.
- `langfuse-public-key`, `langfuse-secret-key` — Langfuse OTLP exporter credentials.
- `neon-database-url` — pgvector connection string for the agent.
- `cloudfront-agent-secret` — CloudFront-injected shared-secret header that gates the agent Function URL.

Until now these lived in AWS Secrets Manager under `/fpl-platform/{env}/*`. AWS Cost Explorer for April 2026 showed Secrets Manager as 70% of the platform's monthly AWS spend ($1.43 of a $2.02 total) — driven entirely by the flat $0.40/secret/month charge. The platform doesn't use any Secrets Manager feature beyond "store a string and let a Lambda read it": no rotation, no resource policies, no replication, no secret-recovery window we actually rely on.

## Options Considered

### 1. Keep Secrets Manager (rejected)
Status quo. Pay $2.00/month flat ($0.40 × 5 secrets) plus negligible API charges.

Rejected because: every feature Secrets Manager bills for over Parameter Store is a feature we don't use. The $2/month is dominated by the rotation/audit machinery we'd never invoke. Rotation here is `terraform taint` + manual paste from each provider's dashboard; recovery is "regenerate it" because every value is reproducible from its source of truth (Anthropic console, Langfuse project, Neon dashboard, `random_password`). Paying for managed rotation that no caller schedules is a marker of unconsidered spend.

### 2. SSM Parameter Store, SecureString tier (chosen)
SecureString parameters under `/fpl-platform/{env}/*`, encrypted with the AWS-managed key `alias/aws/ssm`.

Standard-tier SecureString parameters are free up to 10,000 per account. The default `alias/aws/ssm` KMS key is also free — its built-in policy auto-grants `kms:Decrypt` to any principal in the account when the call comes via SSM (`kms:ViaService = ssm.{region}.amazonaws.com`). Net cost: $0.00 for 5 parameters where Secrets Manager charged $2.00.

### 3. Plaintext env vars in Terraform/Lambda (rejected)
Pass values as Lambda environment variables, sourced from `terraform.tfvars`.

Rejected because: secret values land in plaintext in Terraform state and in Lambda's `Environment.Variables` block — visible to anyone with `lambda:GetFunctionConfiguration`. The IAM blast radius for "read a Lambda env var" is much wider than for "decrypt a SecureString parameter". Encryption at rest with a documented IAM gate is the bar.

### 4. HashiCorp Vault / Doppler / 1Password Connect (rejected)
External secret managers with proper rotation and audit.

Rejected because: introducing a non-AWS dependency for five credentials at zero scale is an unforced complexity tax. The audit and rotation features are real but not load-bearing here; revisit if the platform ever hosts another team's credentials or hits a compliance requirement.

## Decision

Use SSM Parameter Store SecureString parameters, encrypted with `alias/aws/ssm`, under the existing `/fpl-platform/{env}/*` path. Lambdas continue to fetch values at cold-start via `fpl_lib.secrets.resolve_secret_to_env`; the helper switches its boto3 client from `secretsmanager` to `ssm` and the API from `get_secret_value(SecretId=...)` to `get_parameter(Name=..., WithDecryption=True)`. The path convention and resolver call-sites are unchanged.

The `cloudfront-agent-secret` value continues to be generated by `random_password` and tracked in Terraform state — same pattern, different parameter type. The other four parameters are declared with placeholder values and `lifecycle { ignore_changes = [value] }`, matching the previous "secret shells, populate via Console" workflow.

The Lambda IAM role swaps `secretsmanager:GetSecretValue` for `ssm:GetParameter` + `ssm:GetParameters`, scoped to the same path prefix. No explicit `kms:Decrypt` is needed for the AWS-managed `aws/ssm` key — its key policy grants Decrypt to any principal in the account when called via SSM, which is exactly the path SecureString reads take.

## Cost outcome

| Resource | Before | After | Saving |
|---|---|---|---|
| 5 × Secrets Manager secrets | $2.00/month flat | — | $2.00/month |
| 5 × SSM SecureString parameters | — | $0.00/month | — |
| API calls | negligible | negligible | — |
| **Total** | **~$2.00** | **$0.00** | **~$2.00/month** |

April 2026's actual Secrets Manager bill was $1.43, reflecting partial-month proration on more recent secrets. The full annual saving once amortised is ~$24.

## Trigger to revisit
Switch back to Secrets Manager — or supplement it — if any of these become true:

- A compliance requirement mandates automatic rotation (Anthropic / Langfuse don't support short-lived keys, but Neon roles do, and a future stakeholder might require 30-day rotation).
- Cross-account access to a secret is needed (Secrets Manager supports resource policies; Parameter Store does not).
- The parameter count crosses ~50 in a way that values rotation tooling, audit trails, or fine-grained resource policies enough to pay the per-parameter charge.
- Any single parameter's value exceeds 4 KB (Parameter Store standard-tier limit; advanced tier is 8 KB at $0.05/parameter/month, which approaches Secrets Manager's pricing anyway).

## Consequences
**Easier:**
- AWS bill drops to ~$0.60/month (ECR storage + S3), eliminating the largest single line.
- Same path convention, same `resolve_secret_to_env` call sites — the change is local to one helper and one IAM policy.
- Standard-tier parameters scale to thousands at zero cost; future credentials added under the same path inherit the IAM grant for free.

**Harder:**
- No automatic rotation. If any provider starts requiring short-lived credentials, we'd add rotation tooling separately or move that single secret back to Secrets Manager.
- No native resource policies on parameters; cross-account sharing would require either an assume-role hop or a Secrets Manager replica.
- Parameter Store has a 4 KB SecureString value limit on the standard tier. None of our current values come close (the longest is the Neon connection string at ~120 chars).
- Migration is destructive at the AWS level: Secrets Manager secrets are deleted (`recovery_window_in_days = 0` was already set on `cloudfront-agent-secret`; the others use the default 30-day window and will be schedule-deleted), and the four user-supplied parameters need their values re-pasted into the Console after the apply. Documented in the migration PR and in `docs/runbook.md`.
